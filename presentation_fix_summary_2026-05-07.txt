สรุปสิ่งที่ต้องแก้ไขหลังพรีเซนต์โปรเจค 7-5-2026
โปรเจค: Chiang Rai Photographer Booking

ภาพรวมระบบตอนนี้
1. ระบบเป็นเว็บค้นหาและจองช่างภาพเชียงราย
2. มีผู้ใช้ 3 กลุ่มหลัก: ผู้เยี่ยมชม/ลูกค้า, ช่างภาพ, แอดมิน
3. ผู้เยี่ยมชมดูหน้าแรก, ค้นหาช่างภาพ, ดูโปรไฟล์, ดูบทความ, FAQ, ติดต่อได้
4. ลูกค้าที่สมัครแล้วสามารถจองงาน, ดูประวัติจอง, รีวิว, ดูรายการโปรด, ดูช่างภาพที่เคยดู
5. ช่างภาพจัดการโปรไฟล์, พื้นที่รับงาน, ประเภทงาน, ผลงาน, วันว่าง, การจอง, บทความ, รีวิว
6. แอดมินจัดการผู้ใช้, ช่างภาพ, หมวดหมู่, อำเภอ, การจอง, รีวิว, บทความ, แบนเนอร์, รายงานปัญหา, ข้อความติดต่อ, ประวัติระบบ
7. ระบบมีฐานข้อมูลรองรับ search_logs, recently_viewed, reports, notifications, activity_logs, booking_status_logs แล้ว
8. จุดที่อาจารย์ติงส่วนใหญ่เป็นเรื่อง UX, คำอธิบาย, การจัดกลุ่มเมนู, ภาษาที่ไม่ชัด, และบางฟีเจอร์ยังไม่ครบ

งานที่ต้องทำ Demo
1. เตรียม demo flow ผู้เยี่ยมชม
   - เข้าเว็บโดยไม่สมัคร
   - ค้นหาช่างภาพตามอำเภอ/ประเภทงาน/วันที่ต้องการจ้าง
   - กดการ์ดช่างภาพเพื่อดูโปรไฟล์
   - ดูบทความจากระบบและบทความจากช่างภาพ
2. เตรียม demo flow ลูกค้า
   - login ลูกค้า
   - จองช่างภาพ
   - ดูรายการจองที่ยังดำเนินการอยู่
   - ดูประวัติจองที่เสร็จสิ้น
   - รีวิวงานที่เสร็จแล้ว
   - รายงานรีวิว/ช่างภาพที่ไม่เหมาะสม
3. เตรียม demo flow ช่างภาพ
   - login ช่างภาพ
   - ตั้งวันรับงานผ่านปฏิทิน
   - เพิ่มประเภทงาน/พื้นที่รับงาน
   - รับ/ปฏิเสธคำขอจอง
   - ดูประวัติสถานะงาน
   - เพิ่ม/แก้ไขบทความ
4. เตรียม demo flow แอดมิน
   - login แอดมิน
   - อนุมัติช่างภาพ
   - ดู/กรองคำขอจอง
   - จัดการบทความ
   - จัดการรายงานปัญหา
   - ดูประวัติการใช้งานและอธิบายว่าเก็บจาก activity_logs

รายการแก้ไขฝั่ง Public / ผู้เยี่ยมชม
1. วันจองควรเป็นปฏิทิน
   - ตอนนี้: ใช้ input วันที่แบบข้อความ พ.ศ. ใน photographers.php, customer/create_booking.php, photographer/availability.php
   - ต้องแก้: เปลี่ยนเป็น calendar UI
   - สีที่ควรใช้: ขาว=ว่าง, เทา=ไม่ว่าง, แดง=ถูกจอง, เหลือง=รอตอบรับ
   - ต้องทำให้สื่อว่าเป็น "วันที่ต้องการจ้าง" ไม่ใช่ "วันที่ว่าง"

2. Guest ต้องดูข้อมูลทั่วไปได้โดยไม่สมัคร
   - ตอนนี้: หน้า public หลักดูได้อยู่แล้ว เช่น index.php, photographers.php, photographer_detail.php, blog.php, faq.php, contact.php
   - ต้องเช็คเพิ่ม: ปุ่มที่ควรกดดูได้ต้องไม่เด้ง login ถ้าเป็นแค่ดูข้อมูล
   - เฉพาะการจอง/รีวิว/รายการโปรดค่อยบังคับ login

3. A2 เอกสารต้องเขียนให้เคลียร์
   - ต้องแก้ในเอกสารนำเสนอ/รายงาน ไม่ใช่แค่โค้ด
   - เขียนให้ชัดว่าใครทำอะไร, ข้อมูลเก็บตารางไหน, flow การจองเป็นอย่างไร

4. ตัวหนังสือเล็กในมุมมองเว็บแอพ
   - ตอนนี้: หลายหน้ามี text-sm/text-xs เยอะ โดยเฉพาะ dashboard/table/card
   - ต้องแก้: เพิ่ม font desktop ให้ใหญ่ขึ้น
   - เป้าหมาย: desktop อ่านง่ายสำหรับผู้สูงอายุ, mobile ยังไม่พัง

5. ราคาเริ่มต้น
   - ตอนนี้: photographer_profiles.starting_price และ photographer_services.starting_price มีอยู่แล้ว
   - ต้องแก้: label ให้ชัดว่า "ราคาเริ่มต้นโดยประมาณ"
   - เพิ่มหน่วยบาททุกจุด
   - อธิบายว่าเว็บไม่รับชำระเงิน ราคาไปตกลงกับช่างภาพ

6. กล่องสรุปด้านล่างดูเหมือนปุ่มและทำให้เข้าใจผิด
   - ตอนนี้: หลายหน้าใช้ card/chip/pill เยอะจนเหมือนปุ่ม
   - ต้องแก้: กล่องที่เป็นข้อมูลอย่างเดียวไม่ควร hover/click style
   - label ต้องชัด เช่น "จำนวนช่างภาพ", "คะแนนขั้นต่ำ", "วันที่ต้องการจ้าง"

7. ช่างภาพใกล้เคียงที่แนะนำ
   - ตอนนี้: photographers.php แนะนำใกล้เคียงเฉพาะกรณีไม่พบผลลัพธ์ และใช้ Haversine จาก latitude/longitude ของอำเภอ
   - ปัญหา: ยังไม่ชัดว่าใช้เงื่อนไขอำเภอ/ประเภทงานอย่างไร
   - ต้องแก้: แสดงอำเภอที่เลือกให้ชัด
   - ต้องแก้: แนะนำใกล้เคียงต้องล็อกตามประเภทงานที่เลือกด้วย
   - ต้องอธิบาย: ระยะทางคือระยะทางประมาณจากพิกัดอำเภอ ไม่ใช่ระยะทางถนนจริง

8. การ์ดช่างภาพควรกดได้ทั้งการ์ด
   - ตอนนี้: บางจุดต้องกดปุ่ม "ดู" หรือ clean_context_button
   - ต้องแก้: ทำให้คลิกบนการ์ดช่างภาพได้เลย แต่ไม่ชนกับปุ่มย่อย

9. สีปุ่มต้องมีมาตรฐาน
   - ตอนนี้: มีปุ่มดำ แดง เขียว เหลือง ฟ้า หลายบริบท
   - ต้องแก้: กำหนดมาตรฐานสี
   - เสนอ: ดำ=ดูรายละเอียด/ปุ่มหลัก, แดง=ยืนยัน/CTA, เขียว=อนุมัติ/สำเร็จ, เหลือง=แก้ไข/รอดำเนินการ, เทา=ยกเลิก/ปิด

10. คำว่า "คนบันทึก"
   - ต้องดูว่าหน้าไหนใช้คำนี้
   - ถ้าเป็น activity/admin note ให้เปลี่ยนเป็น "ผู้ดำเนินการ" หรือ "บันทึกโดย"

11. เหตุผลในการรายงานต้องพิมพ์ได้
   - ตอนนี้: database มี reports.reason และ reports.detail
   - ต้องเช็คหน้า user ว่ามี input ให้กรอกเหตุผลเองครบไหม
   - ต้องแก้: เพิ่มช่องพิมพ์เหตุผล/รายละเอียดในหน้ารายงาน

12. คะแนนกับรีวิวไม่ใช่เรื่องเดียวกัน
   - ตอนนี้: reviews มี rating_overall และ rating รายหมวด เป็น TINYINT 1-5
   - คะแนนเฉลี่ยแสดง .1 ได้ แต่ตอนกรอกให้แค่จำนวนเต็ม
   - ต้องตัดสินใจ: จะให้ลูกค้าให้ .5 ได้ไหม
   - ถ้าให้ .5 ต้องแก้ schema จาก TINYINT เป็น DECIMAL(2,1)
   - ต้องแยกคำว่า "คะแนน" กับ "จำนวนรีวิว" ให้ชัดทุกหน้า

13. งานที่สำเร็จดูได้ไหม
   - ตอนนี้: bookings มี status completed และ completed_at
   - ลูกค้า/ช่างภาพ/แอดมินดูได้จากรายการจอง แต่ยังไม่แยกหน้าเฉพาะชัด
   - ต้องแก้: แยกแท็บ "กำลังดำเนินการ" และ "เสร็จสิ้นแล้ว"

14. อัตราการตอบกลับ
   - ตอนนี้: photographer_profiles มี response_rate, average_response_hours และ dashboard แสดง
   - ปัญหา: ผู้ใช้ไม่รู้ว่าคำนวณจากอะไร
   - ต้องแก้: เพิ่มคำอธิบายสั้น ๆ ว่า "คิดจากคำขอที่ช่างภาพตอบรับหรือปฏิเสธ เทียบกับคำขอทั้งหมด"
   - หรือถ้าไม่ช่วย UX ให้ซ่อนจากหน้า public

15. ผลงานดูเป็นศิลปินไปไหม
   - ตอนนี้ใช้คำว่า "ผลงาน" และ Portfolio
   - ต้องแก้ wording ให้ตรงงานจ้าง เช่น "ตัวอย่างงานถ่ายภาพ", "อัลบั้มตัวอย่างงาน"

16. ลบภาพต้องลบไฟล์จริง
   - ตอนนี้: photographer/portfolio.php ลบแบบ soft delete เฉพาะ database deleted_at
   - ไฟล์จริงยังอยู่ใน assets/uploads/portfolios
   - ต้องแก้: ตอนลบ portfolio ให้ unlink ไฟล์จริงด้วย
   - ต้องระวัง: ถ้าเป็น seed image หรือไฟล์ใช้ร่วมกันห้ามลบมั่ว

17. เรียงลำดับในโปรไฟล์ใหม่
   - ตอนนี้: photographer_detail.php มีข้อมูลหลายส่วนในหน้าเดียว
   - ต้องแก้: จัดลำดับให้ชัด เช่น รูป/ชื่อ/ราคา/พื้นที่ -> ประเภทงาน -> ปฏิทินจ้าง -> ผลงาน -> รีวิว -> บทความ -> ติดต่อ

18. เพิ่มประเภทงานให้ชัดเจน
   - ตอนนี้: service_categories มีแล้ว และช่างภาพเลือกใน photographer/services.php
   - ต้องแก้ UI ให้ชัดว่าเพิ่ม/เปิดปิด/รายละเอียดของประเภทงาน
   - แอดมินควรเพิ่มหมวดหมู่งานได้เข้าใจง่าย

19. การแยกพื้นที่จอง/ติดต่อ/ข้อมูลอื่นไม่ชัด
   - ต้องแก้หน้าโปรไฟล์และหน้าจอง
   - แยก section เป็น "ข้อมูลช่างภาพ", "พื้นที่รับงาน", "วันที่ต้องการจ้าง", "ช่องทางติดต่อ", "รายละเอียดคำขอจอง"

20. ดีไซน์หน้าบทความ
   - ตอนนี้: blog.php เป็นบทความระบบ, photographer_articles อยู่หน้าแรก/โปรไฟล์, และเพิ่งเพิ่ม article_detail.php
   - ปัญหา: บทความระบบกับบทความช่างภาพแยกคนละที่ ทำให้หาไม่เจอ
   - ต้องแก้: รวมหน้า "บทความ" ให้เห็นทั้ง 2 แหล่ง
   - เพิ่ม filter: ทั้งหมด / จากระบบ / จากช่างภาพ
   - ใช้คำว่า "บทความ" ให้เหมือนกันทั้งระบบ ไม่ใช้ Blog ปน
   - ถ้าเยอะให้แบ่งหน้า pagination และค้นหาด้วยชื่อ/ผู้เขียน/ประเภท

21. ปุ่มค้นหาช่างภาพในหน้าบทความ
   - ต้องเช็คหน้า blog.php/index.php
   - ถ้าไม่เกี่ยวให้ย้ายออกหรือเปลี่ยนเป็นปุ่ม "ดูบทความทั้งหมด"

22. FAQ ต้องอธิบายแหล่งที่มา
   - ตอนนี้: faqs table และ admin/faqs.php ให้แอดมินใส่เอง
   - ต้องอธิบายตอนพรีเซนต์ว่า FAQ คือคำถามที่แอดมินรวบรวมและจัดหมวดหมู่เอง
   - ต้องแก้ UI: เพิ่มปุ่ม "ดูคำถามทั้งหมด" ถ้าหน้าแรกแสดงแค่บางข้อ
   - ถ้า FAQ เยอะให้ทำค้นหา/กรองหมวด
   - กำหนดจำนวนที่หน้าแรก เช่น แสดง 4-6 คำถามล่าสุด/สำคัญ

23. Navbar คำว่า "ติดต่อ"
   - ตอนนี้: contact.php คือการติดต่อผู้ดูแลระบบ/เว็บ
   - ต้องแก้ label เป็น "ติดต่อผู้ดูแลระบบ" หรือ "ติดต่อเว็บไซต์"
   - ในโปรไฟล์ช่างภาพใช้ "ติดต่อช่างภาพ" แยกชัดเจน

รายการแก้ไขฝั่งลูกค้าที่สมัครแล้ว
1. ประวัติการจอง
   - ตอนนี้: customer/bookings.php มีอยู่
   - ต้องแก้: แยก "กำลังดำเนินการ" กับ "เสร็จสิ้นแล้ว"
   - รายการ dashboard ควรโชว์เฉพาะงานที่ยังไม่เสร็จเป็นหลัก

2. ปุ่มตัวกรองใน dashboard ลูกค้ากดแล้วไม่ไป
   - ต้องตรวจ customer/dashboard.php และ customer/bookings.php
   - ถ้าปุ่มเป็นแค่ stat card ต้องทำเป็นลิงก์ไปหน้ารายการพร้อม filter

3. รายการจองควรขึ้นเฉพาะที่ยังไม่เสร็จ
   - ตอนนี้ dashboard ดึง booking ล่าสุดทุกสถานะ LIMIT 8
   - ต้องแก้ query ให้ dashboard แสดง pending/accepted/confirmed ก่อน
   - completed แยกไปหน้า/แท็บประวัติ

4. ช่างภาพที่เคยดูเก็บยังไง
   - ตอนนี้: ตาราง recently_viewed_photographers เก็บ user_id, photographer_id, viewed_at
   - ฟังก์ชัน record_recently_viewed() บันทึกตอนเปิดโปรไฟล์
   - ต้องอธิบายตอนพรีเซนต์: เก็บล่าสุดตามผู้ใช้ login
   - ต้องมีเอกสารอธิบายตารางนี้

5. Dashboard ลูกค้าควรดูประวัติรีวิว/รีพอร์ตได้
   - ตอนนี้: มี review reminder แต่ยังไม่มีเมนูประวัติรีวิวและรายงานชัด
   - ต้องเพิ่มหน้า/section "รีวิวของฉัน" และ "รายงานปัญหาของฉัน"

รายการแก้ไขฝั่งช่างภาพ
1. กราฟคำขอจอง 12 เดือน
   - ตอนนี้: photographer/dashboard.php นับ booking รายเดือนรวมทุกสถานะ
   - ต้องแก้: แยกสี/แท่งตามสถานะ เช่น กำลังดำเนินการ, เสร็จสิ้น, ยกเลิก/ปฏิเสธ

2. ประวัติสถานะต่าง ๆ
   - ตอนนี้: booking_status_logs มีแล้ว และ dashboard แสดงสถานะล่าสุดบางส่วน
   - ต้องแก้: ใน booking detail ควรแสดง timeline ชัดเจนทุกสถานะ

3. รายการจองควรแยกสำเร็จแล้ว/กำลังทำ
   - ต้องแก้ photographer/bookings.php ให้มีแท็บหรือ filter ชัด
   - Dashboard ช่างภาพควรแสดงงาน upcoming/pending ก่อน

4. ใส่สกุลเงิน
   - ต้องใส่ "บาท" ทุกจุดที่แสดงราคา
   - เช่น starting_price, service starting_price, ราคาใน card/search/profile

5. พื้นที่รับงานควรมี select all
   - ตอนนี้: photographer/service_areas.php เลือกพื้นที่ทีละรายการ
   - ต้องแก้: เพิ่ม checkbox "รับงานทุกอำเภอ" หรือปุ่มเลือกทั้งหมด

6. รายละเอียดประเภทงานใส่ได้เท่าไหร่
   - ตอนนี้: photographer_services.description เป็น TEXT
   - ต้องแก้ UI: บอกจำนวนตัวอักษรที่แนะนำ เช่น 300-500 ตัวอักษร
   - ถ้าแสดงในการ์ดให้ตัดด้วย line-clamp และมีดูเพิ่มเติม

7. สถานะเปิด/ปิดควรชัด
   - ตอนนี้: is_available มีใน profile และ status badge บางหน้า
   - ต้องแก้: แสดงเป็น "เปิดรับงาน" / "ปิดรับงาน" พร้อมสีและไอคอน

8. จัดการวันว่างมีสถานะไม่ว่างทำไม
   - ตอนนี้: photographer_availability.status มี available, unavailable, booked
   - ต้องตัดสินใจ UX:
     - ถ้าใช้เฉพาะวันที่รับงาน ให้เหลือ "ว่าง" และลบรายการแทน "ไม่ว่าง"
     - ถ้าต้องแสดงปฏิทินเต็ม ให้ใช้ "ไม่ว่าง" เป็นวันที่ปิดรับงาน
   - ต้องอธิบายให้ชัดใน UI

9. เพิ่มบทความแบบคล้าย Word
   - ตอนนี้: photographer/articles.php เป็น textarea ธรรมดา
   - ต้องแก้: เพิ่ม rich text editor เช่น TinyMCE/Quill
   - ต้องมีวันที่โพสต์, ลำดับใหม่/เก่า, ปุ่มแก้ไข, ปุ่มลบ
   - ตอนนี้ยังไม่มีแก้ไขบทความช่างภาพ
   - ไลก์บทความยังไม่มี ต้องเพิ่มตารางใหม่ถ้าจะทำ

10. รีวิวควรรายงานได้
   - ตอนนี้มี reports table รองรับ target_type review
   - ต้องเพิ่มปุ่ม "รายงานรีวิว" ให้ user/ช่างภาพในหน้าที่แสดงรีวิว

รายการแก้ไขฝั่งแอดมิน
1. อนุมัติช่างภาพควรกรองหัวข้อได้
   - ตอนนี้ admin/photographers.php มีสถานะ approval_status
   - ต้องเช็ค UI ว่ากรอง pending/approved/rejected/suspended ชัดไหม
   - เพิ่มปุ่มกรองด้านบนถ้ายังไม่ชัด

2. แท็กแนะนำขึ้นมาทำไม
   - ตอนนี้มี suggested categories / tags บางจุด
   - ต้องอธิบายว่าเป็น shortcut ช่วยค้นหา
   - ถ้าไม่มีประโยชน์ให้ตัดออกจาก dashboard/admin overview

3. White space รายการจองล่าสุด/รีวิวล่าสุด
   - ตอนนี้ admin/dashboard.php มีรายการจองล่าสุดและรีวิวล่าสุด
   - ต้องปรับ layout: เพิ่มพื้นที่รายการจองล่าสุด ลดพื้นที่รีวิวล่าสุด

4. ช่างภาพเด่นต้องบอกจำนวนรีวิว
   - ตอนนี้ topPhotographers มี average_rating,total_reviews
   - ต้องแสดงแบบ "5.0 คะแนน / 2 รีวิว"
   - อย่าให้เข้าใจว่า 2 คือคนหรืออายุ

5. กิจกรรมล่าสุดมีประโยชน์อะไร
   - ตอนนี้ activity_logs เก็บ action, table_name, record_id, user_id, ip, user_agent, created_at
   - ต้องอธิบาย: ใช้ตรวจสอบว่าใครทำอะไรในระบบ
   - ต้องแก้ UI ให้แสดง action เป็นภาษาไทย และมีกรองผู้ใช้/ประเภท/วันที่

6. กรองในจัดการระบบ refresh หน้าเลยหรือไม่
   - ตอนนี้หลายหน้ากรองด้วย clean_context และ redirect/refresh
   - ต้องตัดสินใจ: ถ้าไม่อยาก refresh ให้ทำ AJAX
   - ถ้ายังใช้ refresh ต้องอธิบายว่าเป็น server-side filter

7. ID ควรเป็นลำดับไหม
   - ตอนนี้หลายตารางใช้ id จาก database
   - ต้องแก้หน้าจอ: แสดง "ลำดับ" 1,2,3 ตามรายการ แทน id จริง
   - id จริงเก็บไว้ในรายละเอียด/หลังบ้านเท่านั้น

8. ภาษาในระบบต้องเป็นภาษาเดียวกัน
   - ตอนนี้มีไทยปนอังกฤษ เช่น Blog, Portfolio, Dashboard, Rating, Published
   - ต้องแก้ label เป็นไทยให้มากที่สุด
   - คำหลักควรใช้เหมือนกันทั้งระบบ เช่น "บทความ", "ผลงานตัวอย่าง", "คำขอจอง"

9. ผู้ใช้งานกับช่างภาพ
   - ตอนนี้ users เก็บทุก role และ photographer_profiles แยกข้อมูลช่างภาพ
   - ถ้าเมนูแอดมินแยกช่างภาพแล้ว เมนู "ผู้ใช้งาน" ควรใช้คำว่า "สมาชิก"
   - หรือทำเมนูย่อย: สมาชิกทั้งหมด / ลูกค้า / ช่างภาพ / แอดมิน

10. หมวดหมู่งานจาก 0/1 ควรเป็นเปิด/ปิด
   - ตอนนี้ service_categories.is_active เป็น 0/1
   - ต้องแก้ UI ให้แสดง "เปิดใช้งาน" / "ปิดใช้งาน"

11. คำขอจอง แอดมินควรดูได้อย่างเดียวหรือแก้ได้
   - ตอนนี้ admin/bookings.php น่าจะจัดการ status ได้
   - ต้องตัดสินใจ business rule
   - ถ้าแอดมินไม่ควรยุ่ง ให้เหลือดูรายละเอียด/ประวัติเท่านั้น

12. บทความแอดมิน
   - ตอนนี้ admin/blogs.php และ admin/articles.php แยกกัน
   - ต้องแก้: บอกลำดับ, วันที่โพสต์, ผู้เขียน, แหล่งที่มา
   - ช่องค้นหาต้องบอกค้นจากอะไร เช่น หัวข้อ/ผู้เขียน/สถานะ
   - ใช้วันที่โพสต์ published_at ไม่ใช่แค่ created_at

13. แบนเนอร์คืออะไร
   - ตอนนี้ banners ใช้ภาพ hero/ปุ่มบนหน้าแรก
   - ต้องเพิ่มคำอธิบายใน admin/banners.php ว่า "ใช้จัดภาพโปรโมตหน้าแรก"
   - แสดงตัวอย่างตำแหน่งที่แบนเนอร์จะไปแสดง

14. รายงานปัญหามีหัวข้อแต่ไม่มีให้กด
   - ตอนนี้ admin/reports_moderation.php จัดการรายงานได้
   - ต้องเช็คหน้า user ว่ามีปุ่มสร้างรายงานครบไหม
   - เพิ่ม link จาก admin dashboard ไปหน้ารายงานให้ชัด

15. ข้อความติดต่อควรแยกหรือกรอง
   - ตอนนี้ contact_messages มี status unread/read/replied
   - ต้องแก้ admin/contact_messages.php ให้กรองสถานะ/วันที่/คำค้น

16. ประวัติการใช้งานเก็บจากอะไร
   - ตอนนี้ใช้ activity_logs และฟังก์ชัน log_activity()
   - เก็บ user_id, action, table_name, record_id, ip_address, user_agent, description, created_at
   - ต้องเขียนคำอธิบายในเอกสาร/หน้าจอ

17. แจ้งเตือนใส่มาทำไม
   - ตอนนี้ notifications ใช้แจ้งคำขอจอง, เปลี่ยนสถานะ, รีวิว, อนุมัติช่างภาพ, suspend
   - ต้องอธิบายว่าเป็นศูนย์รวมแจ้งเตือนของผู้ใช้
   - ถ้าหน้าอื่นมีอยู่แล้ว ให้แสดงเฉพาะรายการสำคัญหรือเอาออกจาก dashboard

ลำดับความสำคัญในการแก้
1. สูงมาก
   - ปฏิทินวันจอง/วันรับงาน
   - ปรับตัวหนังสือ desktop ให้ใหญ่ขึ้น
   - รวม/กรองบทความระบบกับบทความช่างภาพ
   - แยกรายการจองกำลังดำเนินการกับเสร็จสิ้น
   - ทำการ์ดช่างภาพคลิกได้
   - แก้คำ/label ให้ชัดและเป็นภาษาไทยเดียวกัน

2. สูง
   - อธิบายช่างภาพใกล้เคียงและล็อกตามประเภทงาน
   - เพิ่มรายงานรีวิว/เหตุผลรายงานแบบพิมพ์ได้
   - แก้ portfolio delete ให้ลบไฟล์จริง
   - เพิ่ม edit บทความช่างภาพ
   - เพิ่ม select all พื้นที่รับงาน
   - เพิ่มหน่วยบาทและ wording ราคา

3. กลาง
   - ปรับ dashboard แอดมิน whitespace
   - เพิ่ม filter/คำอธิบายใน activity logs, contact messages, reports
   - ปรับ top photographer ให้แสดงคะแนน/จำนวนรีวิวชัด
   - เพิ่มหน้ารีวิวของฉัน/รายงานของฉัน
   - เพิ่มคำอธิบาย FAQ และปุ่มดูเพิ่มเติม

4. ต่ำ/ถ้ามีเวลา
   - rich text editor คล้าย Word
   - ไลก์บทความ
   - AJAX filter ไม่ refresh หน้า
   - คะแนนแบบ .5 ถ้าอาจารย์ยืนยันว่าต้องมี

คำอธิบายสั้น ๆ สำหรับตอบอาจารย์
1. ระบบแนะนำช่างภาพใกล้เคียงใช้พิกัด latitude/longitude ของอำเภอ และคำนวณระยะทางด้วย Haversine Formula เป็นระยะทางประมาณ ไม่ใช่ระยะทางถนนจริง
2. ช่างภาพที่เคยดูเก็บในตาราง recently_viewed_photographers โดยผูก user_id กับ photographer_id และเวลา viewed_at
3. ประวัติสถานะการจองเก็บใน booking_status_logs ทุกครั้งที่มีการเปลี่ยนสถานะ
4. ประวัติการใช้งานเก็บใน activity_logs ผ่านฟังก์ชัน log_activity() เพื่อดูว่าใครทำอะไร เวลาไหน
5. FAQ เป็นข้อมูลที่แอดมินรวบรวมและจัดการเอง ไม่ได้มาจากคำถามสดของผู้ใช้
6. แบนเนอร์คือภาพ/ข้อความโปรโมตบนหน้าแรก ใช้ควบคุม Hero หรือ CTA
7. แจ้งเตือนใช้รวมเหตุการณ์สำคัญ เช่น คำขอจองใหม่ เปลี่ยนสถานะ รีวิว และการอนุมัติ
8. เว็บไม่ใช่ระบบรับชำระเงิน เป็นตัวกลางค้นหา จอง และติดต่อช่างภาพ
